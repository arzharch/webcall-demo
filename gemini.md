# Project: "Bella Cucina" - Agentic Voice Bot MVP (Phase 2)

## Objective
Re-architect the existing Python/FastAPI voice bot into a streaming, agentic system. The goal is to achieve human-level fluidity (interruptibility, low latency) using a "Brain/Orchestrator" pattern enabled by LangChain.

## Tech Stack Constraints (Strict)
- **Language:** Python 3.10+
- **Framework:** FastAPI (REST for session setup, WebSocket for audio transport)
- **Orchestration:** LangChain (LCEL - LangChain Expression Language)
- **LLM:** Google Gemini 2.0 Flash (via `google-generativeai`)
- **STT:** Faster-Whisper (Local, running on CPU/INT8)
- **TTS:** Coqui XTTS v2 (Local, streaming mode)
- **Storage:** In-memory Python Dicts or SQLite (No Redis)
- **Vector DB:** FAISS (Local)

## Key Concepts to Emphasize

When generating the code, adhere to these core principles:

- **LangChain Expression Language (LCEL):** The `ConversationOrchestrator`'s logic should be built as a declarative chain using LCEL. Avoid monolithic `if/else` blocks. Instead, use `RunnableBranch` for routing and compose chains using the pipe (`|`) operator. This ensures the logic is modular and transparent.
- **Agentic Tools:** Define all external functions (e.g., `search_menu`, `check_availability`) as LangChain tools using the `@tool` decorator. Ensure their docstrings are descriptive, as the Gemini model will use this information to determine when and how to call them.
- **FastAPI Dependency Injection:** Do not instantiate services like the `ConversationOrchestrator` or `Settings` globally. Instead, create them via functions and use FastAPI's `Depends` system to inject them into the WebSocket endpoint. This is critical for managing state and simplifying testing.
- **Pydantic for Data Integrity:** All data structures passed between components (e.g., `SessionState`, API responses) must be defined as Pydantic models. Rely on FastAPI's automatic validation of these models at the API boundary.

## Architectural Requirements

### 1. The "Brain" (agent/orchestrator.py)
Create a class `ConversationOrchestrator` that manages the conversation flow.
- **State Management:** Use a Pydantic model `SessionState` to track:
  - `current_intent` (e.g., "booking", "menu_query")
  - `booking_slot` (date, time, party_size)
  - `conversation_history` (list of messages)
- **Routing:** Implement a "Router Chain" using Gemini and LCEL (`RunnableBranch`). It should classify incoming text into intents: `MENU_SEARCH`, `CHECK_AVAILABILITY`, or `GENERAL_CONVERSATION`.
- **Tooling:** Re-use the logic from `agent/tools.py` but wrap them as LangChain `@tool` decorators.
- **Streaming:** The `process_message` method must be an async generator that yields response chunks as they are generated by the LLM.

### 2. The Audio Pipeline (services/audio_manager.py)
- **STT:** Implement a `TranscriptionService` that accepts audio chunks. It should use a VAD (Voice Activity Detection) logic (like `silero-vad` or energy threshold) to determine when the user has *stopped* speaking to trigger the orchestrator.
- **TTS:** Implement a `SynthesisService` that accepts a text iterator. It must buffer text until a full sentence is formed (using punctuation splitting), then send that sentence to Coqui TTS to generate audio bytes immediately.

### 3. The API Layer (main.py)
- **REST Endpoint:** `POST /session/start` -> Returns a `session_id` and initial context.
- **WebSocket Endpoint:** `WS /ws/audio/{session_id}`
  - Receives: Binary audio chunks (PCM 16k).
  - Sends: Binary audio chunks (Response).
  - The WebSocket loop must handle "Interruption": If the user starts speaking while the bot is sending audio, the bot must stop sending immediately and clear the buffer.

### 4. Data Source
- Use the existing `data/restaurant_kb.json` structure for the menu and FAQs.
- Use `data/tickets.json` for storing reservations.
